    Documentazione tecnica

    Per far partire il server in locale, dalla cartella illy-refactor eseguire il comando npm start

    Alberatura progetto

    Il codice va inserito nella cartella src:
    • nella cartella materials si inseriscono i componenti html (codicecomponente-nomeComponente.html), se si creano sottocartelle il file html si deve chiamare come la sottocartella
    • nella cartella assets -> toolkit -> scripts inserire il file js associato al componente (nomeComponente.js)
    • nella cartella assets -> toolkit -> styles inserire il file scss associato al componente (_nomeComponente.scss)
    • nella cartella pages creare la pagina di visualizzazione del componente referenziandolo come doppia graffa + > nomeComponente + doppia graffa
    • Il path per le immagini da usare nell'HTML è ../assets/toolkit/images/[img]

    Coding guidelines

    Ogni componente va implementato secondo delle guidelines.

    Inoltre, nelle fasi di "pausa" del progetto, ci sono una serie di attività di otttimizzazione che bisognerebbe cercare di implementare:

    • eliminare con attenzione codice e files non più utilizzati (HTML/CSS/JS)--> (fare regolarmente)
    • definire e implementare naming convention per files, classi CSS (BEM), id, funzioni, ecc...
    • individuare design patterns riutilizzabili (atomici)
    • implementare CSS mobile-first: il default (senza media query) è il mobile, mentre tablet e desktop vanno racchiusi in media queries
    • nidificare le media queries all'interno delle singole classi CSS invece che fare dei blocchi per screen size
    • implementare funzionalità SASS:
      • implementare componenti atomici tramite partials
      • nidificare e indentare il codice in maniera uniforme
      • implementare utilizzo variabili (colori, ecc...) --> (fatto colori)
      • implementare utilizzo classi placeholder per evitare ripetizioni di codice
      • implementare utilizzo mixins (media queries, browser prefix, ecc...)
    • utilizzare modernizr per IE11 ???
    • passare da jQuery a vanilla Javascript per non caricare la libreria ???

    Branch guidelines

    1. Una volta avuto indicazione dai funzionali sul tipo di sviluppo e sulla versione dove dovrà essere sviluppata la modifica, se non esiste già, creare da develop un nuovo branch toolkit/v[VERSIONE-SCELTA-DAI-FUNZIONALI] secondo la nomenclatura indicata sotto e fare push
      • master è il branch di produzione
      • develop è il branch di sviluppo
      • toolkit/(major) è la versione del toolkit da portare in produzione (staccata da e mergiata in develop)
      • toolkit/(minor) è la versione del toolkit relativa alle nuove attività da sviluppare definite in Trello (staccata da e mergiata in develop)
      • toolkit/(patch) è la versione del toolkit relativa alle hotfix da sviluppare definite in Trello (staccata da e mergiata in master)
      • feature/nomeComponente_descrizione-feature sono i branch su cui sviluppare nuovi componenti/funzionalità (staccati da e mergiati in toolkit/(minor))
      • bugfix/nomeComponente_descrizione-bug sono i branch su cui fixare nuovi componenti/funzionalità (staccati da e mergiati in toolkit/(minor))
      • refactor/nomeComponente[-descrizione-funzionalità] sono i branch su cui rifattorizzare componenti/funzionalità (staccati da e mergiati in toolkit/(minor))
      • internal/descrizione-funzionalità sono i branch su cui implementare attività di admin/operative che non impattano sullo sviluppo (staccati da e mergiati in develop)
      • hotfix/nomeComponente_descrizione-bug sono i branch su cui fixare componenti/funzionalità di produzione (staccati da e mergiati in toolkit/(patch))
    2. Inviare email a sviluppatori front-end comunicando il nuovo branch dove fare le pr relative a quello sviluppo
    3. Dopo la ricezione dell'email di nuovo rilascio, nel caso di branch di lavoro ancora aperta, fare merge di develop sul proprio branch di lavoro
    4. Quando si termina l'attività sul proprio branch di lavoro, eseguire push -d dello stesso
    5. Per cancellare TUTTI i branch in locale, eseguire git branch | grep -v '^*' | xargs git branch -D. Attenzione: assicurarsi prima di aver pushato su remoto tutti i branch locali di interesse

    Pull request guidelines (sul repository statico Bitbucket)

    Per permettere agli sviluppatori AEM di capire se nelle nostre push ci sono delle modifiche HTML che impattano la loro integrazione piuttosto che solo CSS/JS che invece non ha impatto, suggerisco di usare un messaggio standard da inserire come Title della pull request, tipo così:

    Idticket: attività svolta (HTML/CSS/JS)

    Per esempio:
    • se si è modificato solo il CSS -> ID 36 - APP COMPONENT: casistica solo bottoni (CSS)
    • se si è modificato HTML e CSS -> ID 36 - APP COMPONENT: casistica solo bottoni (HTML/CSS)
    • se si è modificato HTML, CSS e JS -> ID 36 - APP COMPONENT: casistica solo bottoni (HTML/CSS/JS)
    e così via per tutte le casistiche
